## 0. 결론

이 작업은 단순 버그 묶음 수정으로 처리하면 안 된다.

현재 오류 대부분은 개별 UI 문제가 아니라 다음 구조 문제에서 나온다.

1. 도어 H/W/Y 계산이 `DoorModule`, `CADDimensions2D`, `CleanCAD2D`, `calculatePanelDetails`, `usePDFExport`에 흩어져 있다.
2. 선반 들이기, 신발장 깊이, 찬넬 폭, EP 갭 같은 모듈 규칙이 `moduleId.includes(...)` 분기로 반복된다.
3. 2D 화면과 PDF/DXF가 항상 같은 데이터를 쓰는 구조가 아니다.
4. 새 모듈이 추가될 때 렌더, 치수, export 중 하나가 누락될 가능성이 높다.

따라서 작업 순서는 반드시 다음과 같이 간다.

```text
Phase -1  기준 잠금
Phase 0   공통 계산기 관찰 모드 추가
Phase 1   독립 저위험 수정
Phase 2   단일 기능 수정
Phase 3   공통 계산기로 교체
Phase 4   데이터 의미 변경/마이그레이션
```

### 0.2 발주도면 절대 기준

아래 항목은 코드 취향이나 임시 구현으로 바꾸면 안 되는 생산 기준이다.

1. Export는 2D 화면에 보이는 몸통, 서랍, 선반, 옷봉, 상하부 프레임, 백패널, 보강대, 경첩, 도어, 치수를 누락 없이 사용한다.
2. 측면뷰 export는 좌측뷰만 허용한다. 우측뷰는 UI 선택값, 레거시 저장값, PDF 템플릿, DXF 생성 경로 어디에서도 최종 출력되면 안 된다.
3. 측면뷰는 슬롯별로 생성한다. 듀얼 모듈은 점유한 두 슬롯 기준에서 모두 누락 없이 보여야 한다.
4. 도어 도면에는 경첩 표시가 있어야 한다.
5. 도어 높이 치수는 외곽 좌/우에만 표시한다. 도어마다 반복해서 중앙에 높이를 찍지 않는다.
6. 도어 폭 치수는 누락되면 안 된다. 듀얼 도어는 전체 합산 폭이 아니라 각 leaf 폭 기준을 함께 검증한다.
7. 3D 화면에는 발주용 도어 높이 치수가 떠 있으면 안 된다.
8. 신발장/현관장 H 기본 깊이는 380mm 기준을 유지한다. 클릭 전 400mm로 생성되는 상태는 실패다.
9. 도어 90도 열림은 프레임 두께 절반 수준인 약 9mm만 가구 안쪽으로 들어가야 한다.
10. 내보내기 중 UI 상태 변경은 원복되어야 하며, 화면 깜빡임이 남으면 실제 브라우저에서 재검증한다.

### 0.3 완료 정의

완료는 `npm run build` 통과가 아니다. 아래 조건을 모두 만족해야 한다.

- 대표 fixture 기준으로 3D, 2D 정면, 2D 평면, 2D 좌측, 도어 도면이 같은 치수 기준을 사용한다.
- PDF export와 DXF export가 2D 화면과 같은 모듈 집합을 출력한다.
- export 직후 UI view mode, selected slot, guide/dimension 표시 상태가 원래대로 돌아온다.
- 도어, 신발장, 옷봉, 슬롯별 측면뷰, 패널 목록 회귀 테스트가 있다.
- 실제 브라우저에서 PDF/DXF 파일을 생성하고 샘플 출력물을 확인했다.

## 1. 절대 원칙

### 1.1 큰 리팩토링 금지

한 번에 `DoorModule`, `CADDimensions2D`, `calculatePanelDetails`, PDF export를 모두 바꾸지 않는다.  
먼저 현재 정상 출력 기준을 고정하고, 새 계산기는 기존 결과와 비교하는 관찰 모드로 붙인다.

### 1.2 기준 출력 없이는 수정 금지

다음 항목은 작업 전 기준값을 저장해야 한다.

- 3D 화면
- 2D 정면뷰
- 2D 평면뷰
- 2D 좌측뷰
- 도어 도면
- PDF 출력
- DXF 출력
- 패널 목록/컷리스트

### 1.3 측면도 정책

발주/export 측면도는 **좌측뷰만** 사용한다.  
우측뷰 UI/템플릿/export 경로가 남아 있으면 제거하거나 비활성화한다.

### 1.4 커밋 단위

커밋은 다음 단위로 끊는다.

- 기준 테스트/스냅샷 추가
- 공통 계산기 추가
- 호출부 교체 1개 축
- 버그 수정 1건 또는 강하게 연관된 묶음

### 1.5 수정 순서 원칙

한 기능은 아래 순서로만 완료 처리한다.

1. 기존/목표 기준을 테스트 또는 fixture로 고정한다.
2. 순수 계산기를 만든다.
3. 기존 호출부에 연결한다.
4. 관련 export 경로까지 같은 계산기를 쓰게 한다.
5. 실제 브라우저에서 PDF/DXF 결과를 확인한다.

4번 또는 5번이 빠진 항목은 “부분 연결”이지 완료가 아니다.

## 13. 실제 프로젝트 검증 절차

`/demo` 샘플 검증은 통과했지만, `/demo` Header 컨버팅 버튼은 기업회원 안내 모달로 차단된다. 완료 판정에는 인증된 실제 프로젝트의 Header/ConvertModal 버튼 경로와 사용자 PDF 템플릿 출력 검증이 필요하다.

### 13.1 Header/ConvertModal 버튼 경로

검증 대상 경로:

```text
Header export button
→ Configurator onExportPDF
→ setIsConvertModalOpen(true)
→ ConvertModal
→ PDF 다운로드 / DXF ZIP 다운로드
```

관련 파일:

| 역할 | 파일 |
|---|---|
| Header export 버튼 | `src/editor/Configurator/components/Header.tsx` |
| Header callback 연결 | `src/editor/Configurator/index.tsx` |
| PDF/DXF 모달 | `src/editor/Configurator/components/ConvertModal.tsx` |
| PDF 생성 | `src/editor/shared/utils/dxfToPdf.ts` |
| DXF 생성 | `src/editor/shared/hooks/useDXFExport.ts`, `src/editor/shared/utils/dxfDataRenderer.ts` |

### 13.2 실제 프로젝트 샘플 절차

1. 인증된 계정으로 실제 프로젝트를 연다.
2. 2D 정면/평면/좌측에서 몸통, 도어, 서랍, 백패널, 옷봉, 프레임, 치수가 화면에 보이는지 먼저 확인한다.
3. Header의 내보내기 버튼으로 ConvertModal을 연다.
4. PDF 탭에서 정면, 평면, 좌측, 도어 도면을 포함해 다운로드한다.
5. DXF 탭에서 정면, 평면, 측면, 도어 도면을 포함해 ZIP 또는 통합 DXF를 다운로드한다.
6. 다운로드 후 화면이 원래 view mode, selected slot, 치수/가이드 표시 상태로 돌아왔는지 확인한다.
7. 생성된 PDF/DXF를 열어 아래 합격 기준과 대조한다.

### 13.3 합격 기준

| 항목 | 합격 기준 |
|---|---|
| 우측뷰 | PDF/DXF 어디에도 `Right View`, `right`, 우측뷰 페이지/제목이 없어야 한다 |
| 측면뷰 | 좌측뷰만 있고, 슬롯별 측면뷰가 빈 통으로 나오면 실패 |
| 도어 도면 | 경첩, 도어 외곽, leaf 폭, 외곽 좌/우 높이 치수가 있어야 한다 |
| 도어 치수 | 높이 치수가 각 도어 중앙에 반복되면 실패, 폭 치수 누락도 실패 |
| 듀얼 도어 | 전체 합산 폭만 나오면 실패, leaf 폭 기준이 확인되어야 한다 |
| 현관장/신발장 H | 신규 생성/배치/export 모두 380mm 기준이어야 하며 400mm로 나오면 실패 |
| 옷봉 | 2D 화면과 같은 옷봉/브라켓 위치가 PDF/DXF에 있어야 한다 |
| 서랍/백패널/프레임 | 2D 화면에 보이는 서랍, 백패널, 상하부 프레임, 보강대가 누락되면 실패 |
| UI 상태 | export 후 화면이 다른 뷰나 다른 슬롯으로 남아 있으면 실패 |
| 템플릿 | 사용자 PDF 템플릿에서 우측뷰 슬롯/라벨이 재등장하면 실패 |

### 13.4 산출물 보관

실제 프로젝트 검증 시 아래 파일을 함께 보관한다.

- 생성된 PDF
- 생성된 DXF 또는 ZIP
- PDF 텍스트 추출 결과 또는 주요 페이지 스크린샷
- DXF 레이어/텍스트 검사 결과
- export 전후 UI 상태 메모

이 산출물이 없으면 완료 처리하지 않는다.
